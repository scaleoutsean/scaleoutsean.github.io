    The shocking truth about SANtricity snapshot schedules

    2026-03-21 00:00 3 minute read

    On this page

    • Introduction
    • Snapshot schedules
    • What to do about snapshot schedules?
    • Conclusion

    Introduction

    Over the years I've deep dived into SANtricity snapshots, but only when I needed to and not more than I had to, because it's not a happy affair.

    They work, but … need work.

    There's nothing fundamentally wrong with them. Snapshots aren't exactly a rocket science these days and not much has changed from day one (whenever that was for SANtricity - maybe 20 years ago?).

    On SANtricity, snapshots are Copy-on-Write. You may prefer - or not - some other approach, but that's how it works. It has its advantages and disadvantages (and this isn't a random "glass half empty" statement; SANtricity snapshots do have unique flexibility that is advantageous when used properly).

    The issue is user experience, even when you use the Web UI.

    The experience is much worse when you use the API without the UI, because the UI isn't there to hide "how sausages are made" from you and help you out… There's nothing that couldn't be improved - the main issue is that it's not being improved.

    As I use the SANtricity API a lot lately, so today I came across one noteworthy example: snapshot schedules.

    Snapshot schedules

    The UI makes it nice & easy (see the API if you don't believe me!). For example, it creates snapshot groups (you don't want to know) for you.

    If you don't think about it, it's "good enough" - couple-o-clicks later, you're all set!

    Except… did you notice you can't say "take one once every 30 minutes and keep 4"?

    But, that's not the only problem.

    The second is you can say "take 4 snapshots per day, one every 6 hours", but what that means one snapshot every 6 hours. And what that also means is that, eight days from now, you'll hit the wall (the maximum number of snapshots per snapshot group is 32).

    Then the third problem appears: since there's no "and keep X", even with one snapshot per day, you'll hit the wall in 32 days. So the next thing to worry about is "what then?" and the answer is "check your Snapshot Group, or Snapshot Consistency Group, settings".

    There - since you may have forgotten - you allocate the group (or "repo") size as percentage of base volume (or group of volumes, for Consistency Groups), capacity. There's another setting that says "when we hit the limit, do this", which has two choices:

    • Delete (purge) oldest snapshot (okay, but not really)
    • Make base volume read-only (ouch)

    See the bottom of this screenshot from one of the early posts on SANtricity snapshots:

    Why is purging the oldest snapshot "not really okay"? Because eventually (whether it's 32 days or 36 hours) you will hit the limit.

    But, since snapshots are purged, why is that a problem?

    Well, it's a problem because your volume's snapshot count will be permanently in the state of being maxed out. That means that no one can take a snapshot without first deleting a snapshot.

    Yeah.

    What to do about snapshot schedules?

    Well, it's easy to fix - a better scheduler API is all it takes.

    Since that hasn't happened yet, I've started contemplating my own solution.

    Here's what I'm going to do:

    • Snapshot scheduler with multiple schedules. Nothing spectacular, just the way everyone does it
    • Snapshot monitor. This will be "special" in the sense that others don't need to, or even can't (yep, SolidFire) monitor snapshots. SANtricity snapshot groups and snapshot consistency groups can be monitored, and should be monitored because they're "weakly managed" which is why that screenshot is peppered with comments
    • Runs externally, as a service

    And, once I have this in place, I can start recommending no snapshot schedules on SANtricity, because it's clearly not how it should work.

    So, "how you can fix 'em" is "do not use 'em". But I can't make that recommendation until my (external) snapshot scheduler is out.

    Conclusion

    SANtricity's snapshot scheduler should be improved, but that news is from 2016.

    It has occurred to me, however, that this can be nicely fixed externally. This is unlike the snapshot-related API methods, for example, which can't be.

    And not only can it be fixed, but it can also solve the problem of "the constant state of a maxed out snapshot count".

    What that means is that, once I run my own nicely managed schedules, I also have a clean path to other snapshot-related solutions which SANtricity scheduler API has been blocking. That means the external scheduler should have value and relevance beyond snapshot scheduling, even if the built-in schedule API gets improved.
