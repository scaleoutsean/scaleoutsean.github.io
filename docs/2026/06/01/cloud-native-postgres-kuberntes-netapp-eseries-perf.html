    Introduction

    In the first post, I didn't have time to evaluate performance.

    I still don't, but at least I've done a worst-case scenario test with NVMe/RoCE. This is baseline without any optimization, so it's hard to do worse than this.

    Environment

    • Kubernetes 1.36.1
    • IBM Block CSI driver with SANtricity patch 1.13.2
    • PostgreSQL 18
    • CNPG 1.29.1
    • Host system (for CNPG on Kubernetes and PostgreSQL pgbench on host): dual-CPU Xeon Silver server from last decade
    • EF600 (equivalent of today's entry-level E-Series flash system, EF-Series EF50) with 14 disks in DDP storage pool

    Observations

    How many times have we seen poor performance in such scenarios? I've seen plenty.

    Well, here it wasn't that bad - over 200 MB/s and that's with PostgreSQL client and server sharing the same old box (and yet utilizing just 20% of CPU).

    E-Series' performance monitoring dashboard reflected these dstat figures.

    SELECT easily went over 1.5 GB/s at times.

    Best of all, PostgreSQL didn't break a sweat - most of time it wasn't near the top of the CPU hog list.

    Kubernetes (kube-apiserver, kubelet) was top hog.

    Conclusion

    This "worst case" test yielded great results.

    We haven't even scratched the surface here. We know E-Series can do a lot better with PostgreSQL. But even this casual test shows that - for most users - there's no need to tune anything if you use EF-Series.

    CNPG gives you best practices out of box. It takes 60 seconds to deploy and it comes with HA, backup integration (Velero and Kasten, native backup to S3 (Post Two; currently a TODO item)) and few other things. E-Series gives you fantastic performance without any tuning effort (and not just for Postgres database).
