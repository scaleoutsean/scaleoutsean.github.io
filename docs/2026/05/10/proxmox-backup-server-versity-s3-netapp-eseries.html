    Proxmox Backup Server 4.2, Versity S3 Gateway and NetApp E-Series

    2026-05-10 00:00 6 minute read

    Introduction

    I have a long post on Proxmox Backup Server 4.1 here. This one is about PBS 4.2, focuses on just S3-backed datastores and will be short.

    In Proxmox Backup Server (PBS) 4.2, S3 support graduated from a technology preview to a supported feature.

    S3 users with many hundreds of TBs should probably use S3 appliances such as NetApp StorageGRID. Smaller environments can run StorageGRID in VMs or Versity S3 Gateway in containers, or on bare metal servers.

    In this post we used PBS 4.2 with Versity S3 Gateway v1.4.1. There isn't much to write about, though - it mostly just works.

    Deploy, backup, restore

    • PVE 9.1.9
    • PBS 4.2.0 (with S3 datastore support and Linux kernel 7)
    • Versity S3 Gateway (VGW) v1.4.1

    VGW was deployed on a LUN backed by E-Series. Because we did not evaluate performance (which I knew could not be fully consumed by a 2 vCPU PBS VM - see my PVE post with performance tests), it was just one of existing LUNs.

    When deploying VGW, unlike in recent VGW-themed posts, I specified to use HTTPS and have enabled versioning just to test it out.

    Then I added this S3 to PBS and had no problems setting it up whatsoever. The only annoyance was the usual PBS requirement to get a certificate fingerprint for self-signed TLS certificates while defining S3 endpoint, which I did by examining the VGW certificate from the browser.

    I deployed PBS 4.2 in a VM (good enough for testing), configured a ZFS datastore on the second VM volume backed by E-Series LVM, and a separate S3-backed backup repository. This was so that I can compare the two in case of any issues with the S3-backed backup repository.

    This is an VGW S3 Endpoint in PBS 4.2.0 settings. Notice I use /tmp/ as my temporary directory, which should be some more persistent path.

    That bucket, pbs, viewed in the Versity UI, initially holds just several objects generated when configuring it.

    The same VGW instance is used to host a bucket for Velero (used to backup E-Series CSI data), so you can consolidate your Kubernetes and PVE backups on the same S3 storage.

    An LXC container has been created on a SANtricity LVM device for the purpose of backup/restore testing.

    The first backup to S3 was taken with the container running (PVE handles that easily) and it went smoothly, comparable to backing up to a standard PBS datastore on ZFS (but remember that S3 uses temporary directory in PBS to store data, so traditional filesystem datastores may be more robust).

    I checked the PBS bucket on VGW - new data was in it.

    I didn't RTFM to see if object versioning (which I had enabled on VGW) would be any useful. On the other hand, note that on VGW, object versioning is still experimental, and requires a separate "versioning" path (in addition to data path) when starting VGW service.

    Then I enabled and executed data verification for that backup job, which was successful.

    This is the CT101 data in the PBS S3 datastore pane.

    The same in a larger view, showing job verification was successful and that 500 MB container disk backup file was there.

    In S3 object store, backup data chunks are stored in the chunks subdirectories (<bucket>/<s3_endpoint>/.chunks/[0000-ffff]).

    I went back to PVE 9 and navigated to the CT, Backup > vgw3, to view the CT backups on S3.

    I restored from that initial backup and that went fine.

    One thing I had forgotten, and should mention it, is that it's possible to restore individual files and/or directories.

    That's done by downloading selected file(s) and/or directories. Very nice!

    Side notes

    I initially had glitches resulting from trying to serve S3 over HTTP (so that I can avoid TLS). Maybe it was that, or the funny default service port that VGW uses (7070).

    After switching to HTTPS and registering the certificate fingerprint with PBS (in S3 endpoint configuration), everything worked fine.

    Conclusion

    PBS now has official support for S3 backup stores.

    E-Series has all-flash, hybrid and NL-SAS models that are all suitable for PBS. As discussed in the first PBS post, ZFS (or other) on E-Series NL-SAS is the simplest and least risky option. If you want to use S3, though, you can. It may be easier for those who backup, or replicate backup data, to remote locations and want to protect backups with S3 Object Lock.

    Versity S3 gateway users may want to serve data from a large volume on DDP, especially in NL-SAS environments. A single volume backing VGW can be grown online to hundreds of TBs. StorageGRID SDS users would need more volumes for a more granular storage layout (RTFM). Both Versity Gateway and StorageGRID have versioning and replication to the cloud is possible, making this a potentially easier-to-manage option if replication of backups or backup data is involved.
