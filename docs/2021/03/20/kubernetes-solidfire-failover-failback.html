Introduction

So far I’ve created two videos on the topic of SolidFire replication: one about configuring SolidFire array & volume replication using SolidFire PowerShell Tools in which I demonstrated how SolidFire replication can be configured in mere seconds, and another about manual failover and failback of Kubernetes with SolidFire which used an old approach (before Trident Volume Import feature became available).

The Trident documentation v21.01.01 doesn’t have much on this topic so I’ve been thinking to examine this topic again.

SolidFire replication and switch-over under the hood

How replication works under the hood of SolidFire is simple. Assuming your network configuration is symmetric (that is, your volume X isn’t exposed on VLAN 100 and VLAN 200, respectively), you only need to worry about the following differences between sites:

  • Storage VIP (one per SolidFire cluster) (example: 192.168.1.30)
  • Unique cluster ID (example: mn4y)
  • Volume ID (example: 270)

Of course, there’s also a Volume Name, which can be consistent across sites, prefixed with a unique site-specific or Kubernetes cluster-specific string, or entirely different.

Example:

Cluster SVIP Unique ID Vol Name Vol ID
PROD 192.168.1.30 mn4y pvc-ba229a61-6026-44bd-8b39-29863261469a 270

When an iSCSI client connects to a volume like that, this is what we see in Reporting > iSCSI Sessions in the SolidFire Web UI:

  • iqn.2010-01.com.solidfire:mn4y.pvc-ba229a61-6026-44bd-8b39-29863261469a.270

The same can be observed with Get-SFIscsiSession (PowerShell):

Now we know how to put together our SF iSCSI target:

  • iqn.2010-01.com.solidfire: + ${CLUSTER-UNIQUE-ID} + . + ${VOLUME-NAME} + . + ${VOLUME-ID}

We can get Svip (Storage Virtual IP) and UniqueID using Get-SFClusterInfo, which means we have everything required to transform PROD iSCSI device paths and swap them for Target device paths (which is what storage site failover would require), and back.

This is how I did it in one of the videos mentioned at the top. But the NetApp dynamic provisioner for CSI orchestrators, Trident, has Volume Import feature which has the potential to make site failover a more pleasant experience. Let’s find out if that is true for SolidFire users.

Setup and workflow

There are several different scenarios to consider:

Scenario K8s clusters SF clusters Protection Switch-over
1 1 2 SF cluster SF switch-over
2 2 2 K8s or SF cluster K8s and SF s-o
3 1,3 1,3 AZ No s-o or K8s+SF s-o

In Scenario 1 we assume PROD=>DR replication is in place. There is only one Kubernetes cluster so we just need to switch to the surviving storage back-end.

In Scenario 2 there’s one Kubernetes and one SolidFire cluster per each site. If either cluster fails, we switch both Kubernetes and SolidFire to the surviving site. The difference compared to Scenario 1 is the remote site has a Kubernetes cluster that is unaware of the existence of the PROD site. As soon as we “promote” the remote SolidFire cluster to Read/Write, it can come online for read-write access by the Kubernetes cluster at the remote site.

Scenario 3 is about more complex combinations. One of them (single cluster, three sites scenario) involves SolidFire Protection Domains (PD’s), as well as more than two Kuberntes and SolidFire clusters. What do PD’s do?

They’re similar to Availability Zones in Public Coud. SolidFire cluster with six or more nodes can be logically segregated in three Protection Domains so that two copies of each data block (that SolidFire makes to protect from disk or node failures) never land in the same Protection Domain (in this example, get stored on two nodes belonging to the same PD). If your Kuberetes is spread spread over multiple buildings in a campus environment, you could deploy six or nine SolidFire nodes to protect data and services in the case one building loses power or network connectivity. Only this approach has RTO and RPO of zero, because there is no SolidFire cluster failover (PD failover is just a failover of a node, or domain).

This post will discuss only Scenario 1, because Scenarios 2 and 3 are similar enough.

SolidFire failover

This is my setup:

  • One K8s cluster with Trident CSI
  • Two SolidFire storage clusters paired for volume replication
  • Only one storage back-end (PROD) and one (default) storage class configured in Kubernetes

Plan:

  • Create a PVC, identify resulting PV
  • Configure SolidFire asynchronous replication (PROD=>DR) for that PV
  • Power off the SolidFire array at the PROD site (simulate unplanned site failure)
  • At the DR site
    • On the remote SolidFire cluster: delete volume pairing relationships (I could pause them but I no longer need them)
    • On the remote Kubernetes cluster: delete “stale” PVCs from the PROD site (we need to recreate them on Volume Import)
    • On SolidFire, promote DR replica Volumes to Read/Write (we need to be able to write at the DR site)
    • Add SolidFire back-end to Kubernetes (we pretend our Kubernetes DR cluster is a new cluster)
    • Use Trident Volume Import to import DR site volumes and configuration (PVC YAML) into the Kubernetes cluster at that (DR) site

Replicate this volume to SolidFire on DR site. SolidFire calls this “volume pairing” (more on that in Appendix, if you wish to use the CLI or automate).

Now let’s assume the PROD site failed.

Failover: PROD=>DR

The first thing we’d do is switch to SolidFire UI or API endpoint at the DR site and pause volume replication on that array.

After that we’d delete the stale PVCs and PVs (remnants from the PROD site) on our remote Kubernetes cluster:

$ kubectl delete pvc pg -n pg
persistentvolumeclaim "pg" deleted

$ kubectl delete pv pvc-fdd499f6-14f3-4280-bdef-35d84c5794bb -n pg
warning: deleting cluster-scoped resources, not scoped to the provided namespace
persistentvolume "pvc-fdd499f6-14f3-4280-bdef-35d84c5794bb" deleted

Then we would delete our volume pairing configuration while nothing Volume IDs from both sites (why, because we need to replicate in reverse later, and for that we can’t lose track of Volume IDs required to configure Volume Pairs). You probably don’t want to take notes during failover, but when you pair volumes.

This isn’t “recommended by NetApp” but it is possibly convenient. You could create CSV, JSON or YAML table and keep it in S3 bucket or Github, so that you can always access it from both sites without relying on any notes.

ns/pod src dst
pg/pg 476 126
pg/logs 477 128
pg/bkp 478 130

As you create PVCs for apps that need protection, you can updates that table in the same go. When we pair volumes, we pull/download the table and use it as input to Ansible, PowerShell, or other tool.

We need to promote Target (volume replica in the DR cluster) to Read/Write and we’ll later pair it in reverse (DR=>PROD). As for the PVC/PV at the remote site, we need to recreate them using the DR cluster, and PVs can’t be duplicate. So PVCs must be removed from our Kubernetes cluster. As mentioned earlier, in Scenario 2 (one Kubernetes per each site) we would not have to do this because the remote Kubernetes cluster would not have PV information from the PROD site.

Also recall that because our Storage Class uses Retain, removing the PVCs will still leave us with “stale” PVs which are harmless but can be confusing.

Failback (DR=>PROD)

Remember how we deleted volume pairings at the DR site? The source array isn’t aware of that and when it comes back online it will unsuccessfully attempt to resume replication (PROD=>DR).

So when SolidFire array from the PROD site comes online we can delete its stale volume pairings originally set up for PROD=>DR volume replication and set up new pairings (DR=>PROD) so that we can replicate DR to PROD, and eventually fall back to the PROD site.

We begin like so:

  • On the PROD SolidFire cluster: go to Data Protection > Volume Pairs, and delete stale volume pairs
  • On the same SolidFire cluster: go to each replicated volume and demote it from Read/Write to replicationTarget
  • Return to the SolidFire DR cluster, set up new volume pairs (DR=>PROD) to initiate replication for DR=>PROD failback

Wait until your volume pairs are in sync. If you haven’t forgotten to change the PROD volumes to replicationTarget, you’ll see replication kick off in minutes. SolidFire replication is efficient - only the changed blocks will need to be copied back (and even those will be compressed and deduplicated) - so there is symmetry in failback as far as the amount of data and process are concerned.

Now that your data is ready, you can schedule take the DR site down. Confirm your storage class reclaim policy is Retain, and give up your PVCs at the DR site. After PVCs are deleted, PVs should remain (Retain) in your cluster. (For Scenario 2, because the DR clusters are separate, you wouldn’t need to delete PVCs, you’d just scale to 0.)

$ kubectl delete pvc pg -n pg
persistentvolumeclaim "pg" deleted

$ kubectl get pv -n pg
NAME                                       CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS     CLAIM   STORAGECLASS   REASON   AGE
pvc-d24cf316-496a-4807-aa54-08999f712f9d   1Gi        RWO            Retain           Released   pg/pg   basic                   20m

Now you can stop replication (DR=>PROD) by pausing (or deleting) volume pairing on the SolidFire DR cluster, and then do the same on the PROD SolidFire side.

Reestablish PROD=>DR replication

Promote PROD volumes to Read/Write. Before failback we set volumes at the DR site to replicationTarget and removed volume pairing configuration at both sites, so now we can create volume pairs just as if were starting from scratch (before PROD=>DR failover) and replication will resume.

Now, this step is controversial and not in the official documentation and therefore not recommended either: on my Kubernetes cluster I’d reinstall Trident.

./trident-installer/tridentctl uninstall -n trident
./trident-installer/tridentctl obliviate crd -n trident
./trident-installer/tridentctl install -n trident

Again, in Scenario 2, with one Kubernetes cluster per each site, you wouldn’t have two back-ends on either of the Kubernetes clusters and wouldn’t benefit from doing this step either. But I have one cluster and with that step I think I somewhat simplify my failback.

Summary

Kubernetes failover and failback with Trident and SolidFire are cleaner than the “old school” approach that used patching. Trident’s Volume Import makes the proces much more convenient.

While reinstallation of Trident for failback is not required, I found that approach to work quite nicely and quickly. I don’t know what downsides it might have.

Kuberntes PV names are unsuitable for human consumption so SolidFire’s primary focus on IDs makes it possible to ignore Volume Names and stick to managing Volume IDs. If store Kubernetes configuration and volume pairs in a repository we can use that to automate the creation of remote volumes and Src-Dst pairs and avoid Excel.

Because Trident deletes SolidFire volumes by purging them, it is important to ensure that storage classes for replicated volumes use have reclaim policy for volumes (and possibly volume snapshots, see in Appendix for additional notes on that) set to Retain.

Topology-aware CSI configurations may simplify the management of multiple back-ends in the scenario where one Kubernetes cluster can access two SolidFire back-ends.

Appendix

Pair SolidFire arrays for replication and pair PV with a new PV at the DR site

Source volume accepts writes from iSCSI client(s) and forwards them to Target volume which can only accept writes from the volume replicating to it.

I’ll describe this in more detail because this is easy to automate and lets us almost completely eliminate repetitive work involved in SolidFire failover and failback.

  • Get a list of all volumes you want to pair (e.g. pvc-*)
  • Get properties of each volume (Source and Target)
  • Create Target volume(s) at DR site using the same volume size(s), 512e emulation (on/off) settings, and (optionally) volume name(s)
  • Record Volume ID of each created volume
  • Change access properties of Target volume to replicationTarget
  • With Volume ID pair(s) you gathered in the process configure replication from PROD to DR (sync, async, or snapshots only)

You can decide if you want sync, async or snapshots only. It doesn’t matter which one you choose, as long as it works for you.

SolidFire Volume Names and IDs

Notice how Volume Names in the example above are identical. SolidFire is volume names are just tags. Volume IDs make volumes unique within a cluster, but Volume Names can be duplicate (which can create confusion and so when we create clones or new volumes in the same cluster, we try to avoid that).

We can’t decide Kubernetes volume names when they are created, and we shouldn’t change them once they are created. Across clusters, each volume in a pairs can have the same Name (as in the example above) and some people may find that easier to work with.

Trident has a volume prefix setting, so in Trident back-end files we could ask Trident to prefix volumes at PROD site with prod-, and volumes at DR site with dr-. Because names of volumes at Replication Target site are decided by us when we set up volume pairs we can pick any allowed, but I assume most folks would prefer one of these approaches:

Approach Vol Name at PROD site Vol Name at DR site
Asymmetric prod-pvc-ba229a61-6026-44bd-8b39-29863261469a dr-pvc-ba229a61-6026-44bd-8b39-29863261469a
Symmetric pvc-ba229a61-6026-44bd-8b39-29863261469a pvc-ba229a61-6026-44bd-8b39-29863261469a
Asymmetric for Humans pvc-ba229a61-6026-44bd-8b39-29863261469a pg-data

Kubernetes picks its own volume names, but we pick the names of remote volumes that we create before we set up replication. I’m not at all convinced that naming the remote volume dr-pg-data would be helpful but yes, you could have your own schema ($SITE-$POD-$DISK) for replica volumes at least. I do think, though, it’d be more fruitful to just figure out how to use Git or S3 and automate.

What’s also replicated with Volume is its other tags (called Attributes).

Form the Trident documentation: “When a volume is imported, an annotation is added to the PVC and PV that serves a dual purpose of indicating that the volume was imported and if the PVC and PV are managed. This annotation should not be modified or removed.”

Why use Retain as our Reclaim Policy

You can use Delete, too. If your protected volumes use the retention policy of Delete, you don’t have to deal with Retain’ed PVs. Convenient!

Buuuut, when you release a PVC, the PV gets both deleted and purged. Annnd it’s gone! Yes, you still have a copy at the remote site (at least you had when you last checked three weeks ago), if that’s of any help. You may need to replicate that data back and within hours you’ll recover from that mistake. What you don’t want is to to make two mistakes at the same time (lose a volume after you’ve already lost a site).

Retain is a safer approach to critical data, especially when there’s a lot of it and you can’t instantly re-replicate mistakenly reclaimed volumes.

SolidFire replication and Kubernetes Snapshot Volume Class

For important data, Snapshot Volume Class would usually also be set to Retain. But you’d be unable to delete Snapshots (they’d remain Retain’ed, like PVs with reclaimPolicy set to Retain). Also note that Trident doesn’t include snapshots into replication configuration of underlying volume, so what you could do is:

  • schedule snapshots in SolidFire (crash-consistent snapshots) and make them included in replication. This is easy and very convenient. You don’t get application-consistent snapshots (unless you suspend the app that uses that volume), but you don’t have to deal with Kubernetes.
  • make snapshot replication part of your snapshot worklfow: take a Kubernetes snapshot, find its snapshot ID on SolidFire, and enable its replication with the underlying volume
  • use external application and data protection solution that understands Kubernetes and leverages CSI snapshots to do this for you
