Limit blast radius of CSI-provisioned container cluster with ONTAP storage

20 Feb 2021 - 3 minute read

Problem

“We’d like to automate, but we don’t expose storage management interface to workload nodes”.

Solutions

There are several.

For the maximum security in management, don’t use CSI. Provision manually. Problem solved.

Another way is to use Storage Virtual Machine(s) or SVMs, which are conceptually similar to hypervisor VMs:

  • virtual data network
  • virtual management network
  • virtual management accounts
  • virtual storage (RAID group)

An ONTAP Storage VM (SVM) doesn’t always have to have all these, but it can. To keep the post shorter, let’s say we configure all three. What are the benefits of SVMs for Trident CSI users?

  • Your container cluster (K8s or whatever you have) uses its own SVM
    • Worker nodes use virtual data network(s) to to the SVM’s storage resources (NFS, iSCSI)
    • Because SVM is exposed only to this container cluster, no other clients can get to this data
    • Because SVM “owns” only designated RAID group, it cannot access “host level” (ONTAP Cluster) resources
  • Trident CSI, one of container workloads running on worker nodes, can get to SVM Management IP
    • Trident can use its own account (with minimal privileges) or SVM admin (Vsadmin) account
    • Because worker nodes can access SVM Management IP, you can set up two SVM management accounts: one for Trident (HTTPS; minimum privileges) and one for SVM admin (SSH or HTTPS or SSH+HTTPS; maximum SVM-scoped privileges) and use the SVM admin account for non-CSI purposes such as Ansible
  • ONTAP Cluster
    • Retains its cluster management interfaces which are exposed to a completely different network (highly secure “management LAN”)
    • Has its own management accounts
    • Can have multiple SVMs
    • Can have multiple RAID groups

In an environment with one ONTAP cluster, multiple container clusters could each run on their own Service, SVM Data, and SVM Management VLAN(s). An example for one container cluster with one SVM on one ONTAP cluster:

Who? Corp Mgmt LAN Service VLAN SVM Data VLAN SVM Mgmt VLAN Backup VLAN
K8s Masters   Y   Y Y
K8s Workers   Y Y Y Y
ONTAP SVM     Y Y Y
ONTAP Cluster Y       -

* You could have the Masters wired to SVM Mgmt VLAN for Ansible and whatnot, but you can also run Ansible from a user shell account on Worker nodes.

In my demo (link at the bottom) I cheated in the sense that ONTAP Cluster was in fact connected to External/Service VLAN, but from the demo it is clear that SVM Data and Management LANs are the only way Trident and Workers could use to reach SVM interfaces.

With this setup (multiple, independent) container cluster administrators can automate their storage without impacting anyone else.

They can’t get 100% of ONTAP API functionality this way - because some actions require ONTAP Cluster admin account and access to ONTAP Cluster admin IP - but that’s what security is about.

As hinted above, a container cluster can simultaneously connect to several Trident back-ends, so we can use many available approaches. For example, we configure Trident CSI to use an SVM admin-level account for one SVM (a Trident CSI back-end for which we want more convenience and less security), add a more limited dedicated SVM account for iSCSI on another SVM where we won’t more security and less flexibility, and add yet another SVM which isn’t dynamically provisioned at all.
